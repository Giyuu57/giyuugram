# Giyuugram

A full-featured Instagram clone built with **Flutter + Riverpod + GoRouter** on the frontend and **Supabase** (Postgres, Auth, Storage, Realtime) on the backend.

This zip contains the complete **Dart/Flutter source code**, the **Supabase SQL schema**, and a **web/index.html** snippet — but not a native `android/`, `ios/`, or generated `web/` project shell, since those are toolchain-generated and machine-specific (Gradle/AGP/Xcode versions, SDK paths, signing configs, etc.). You'll generate those with `flutter create` and then drop this source on top, exactly as described below.

Everything below assumes you're starting from zero. Follow it in order.

**Not included** (you generate these locally): `android/`, `ios/`, `macos/`, `windows/`, `linux/`, and the rest of `web/` beyond `index.html`. This is intentional — these folders are produced by `flutter create` based on your installed Flutter/Android Studio/Xcode versions, and copying someone else's generated native folders is a common source of build breakage (we hit exactly this kind of issue with Gradle/AGP versions during development — see the Troubleshooting section).

## Prerequisites

- **Flutter SDK** installed and on your PATH (`flutter --version` should work). This project was built against Flutter's stable channel with Dart SDK `>=3.3.0 <4.0.0`.
- **Android Studio** (for the Android SDK, emulator, and Gradle) if targeting Android.
- **A free Supabase account** — [supabase.com](https://supabase.com)
- **A Google Cloud account** (free) — only needed if you want Google Sign-In working; email/password auth works without it.
- (Optional) A physical Android device or emulator with Google Play Services, since some features (Google Sign-In, camera, video recording) behave more reliably on-device than in an emulator or on web.

## Step 1 — Scaffold a fresh Flutter project and merge this source in

Native platform folders (`android/`, `ios/`, `web/` internals) must be generated by your own Flutter installation — copying someone else's won't reliably build.

```bash
# 1. Create a fresh Flutter project with the same name
flutter create giyuugram
cd giyuugram

# 2. Delete the default generated lib/ and pubspec.yaml
rm -rf lib
rm pubspec.yaml

# 3. Copy everything from this zip's extracted folder into your new project,
#    EXCEPT don't overwrite the android/, ios/, or other native folders
#    flutter create just generated for you.
#    From the extracted giyuugram-source folder, copy:
cp -r /path/to/extracted/giyuugram/lib .
cp /path/to/extracted/giyuugram/pubspec.yaml .
cp -r /path/to/extracted/giyuugram/assets .
cp -r /path/to/extracted/giyuugram/supabase .
cp -r /path/to/extracted/giyuugram/README.md .

# 4. Merge the Google Sign-In meta tag into your generated web/index.html
#    (open both files and copy the <meta name="google-signin-client_id" ...>
#    line from the extracted web/index.html into your generated one's <head>)
```

After this, your project root should have: your freshly-generated `android/`, `ios/`, `web/` (with the meta tag added), plus this zip's `lib/`, `pubspec.yaml`, `assets/`, and `supabase/`.

## Step 2 — Install dependencies

```bash
flutter pub get
```

If you hit a dependency resolution error mentioning `analyzer`, `freezed`, or `custom_lint`/`riverpod_generator` conflicts — this `pubspec.yaml` has already been trimmed to avoid that (we removed `riverpod_generator`/`riverpod_annotation`/`custom_lint` since this codebase uses hand-written Riverpod providers, not code-generated ones). If you add those packages back later, pin compatible versions carefully.

## Step 3 — Generate the freezed/json_serializable model code

The app won't compile without this — it generates `*.freezed.dart` and `*.g.dart` files for every model in `lib/shared_models/` and `lib/features/auth/domain/app_user.dart`.

```bash
flutter pub run build_runner build --delete-conflicting-outputs
```

You should see it report writing outputs for `post.dart`, `comment.dart`, `story.dart`, `activity.dart`, `conversation.dart`, `message.dart`, and `app_user.dart`.

## Step 4 — Set up Supabase

1. Go to [supabase.com](https://supabase.com) → **New Project**. Pick a name, database password, and region.
2. Once the project is ready, go to the **SQL Editor** → **New Query**.
3. Open `supabase/migrations/0001_init.sql` from this zip, copy its entire contents, paste into the SQL Editor, and **Run**.
   - This single file creates every table, index, RLS policy, trigger, storage bucket, and storage policy needed for all features (auth profiles, posts, stories, comments, likes, follows, activities/notifications, direct messages, and file storage for avatars/posts/stories/chat-media).
   - It also enables Supabase Realtime on the `activities` and `messages` tables (required for live notifications and live chat).
4. Go to **Project Settings → API**. Copy your **Project URL** and **anon / public** key (NOT the `service_role` key — that one must never be embedded in client code).

## Step 5 — Plug your Supabase credentials into the app

Open `lib/core/services/supabase_client.dart` and replace the placeholders:

```dart
  ...
}
```

## Step 6 — (Optional) Set up Google Sign-In

Email/password auth works immediately with no extra setup. Google Sign-In requires a bit more:

### 6a. Create OAuth clients in Google Cloud Console

Go to [console.cloud.google.com](https://console.cloud.google.com) → **APIs & Services → Credentials** → **Create Credentials → OAuth Client ID**. Create **two** clients:

**Web application client** (required — used for both Android and Web sign-in verification):
- No extra fields required beyond naming it.
- Under **Authorized JavaScript origins**, add `http://localhost:PORT` for whatever port you'll run `flutter run -d chrome --web-port=PORT` on, if you plan to test on web.
- Copy the generated **Client ID** and **Client Secret**.

**Android client** (required if targeting Android):
- **Package name**: matches `applicationId` in `android/app/build.gradle` (defaults to whatever you named the project when running `flutter create`, typically `com.example.giyuugram` unless changed).
- **SHA-1 certificate fingerprint**: get this by running, from your project root:
  ```bash
  cd android
  ./gradlew signingReport      # macOS/Linux
  .\gradlew.bat signingReport  # Windows PowerShell
  ```
  Copy the `SHA1:` value under the `debug` variant for local development. You'll need to add your **release** keystore's SHA-1 here too before publishing.
- This client has no ID/secret you paste anywhere in code — it just needs to exist, registered with the right package name + SHA-1.

### 6b. Configure the OAuth consent screen

Still in Google Cloud Console → **OAuth consent screen**: fill in an app name and support email, save, and (while in Testing mode) add your own Google account under **Test users** so you can sign in during development.

### 6c. Paste the Web Client ID into three places

1. `lib/features/auth/data/auth_repository.dart`:
   ```dart
   const webClientId = 'YOUR_WEB_CLIENT_ID.apps.googleusercontent.com'; // <-- replace
   ```
2. `web/index.html` (after you've merged it into your generated web folder per Step 1):
   ```html
   <meta name="google-signin-client_id" content="YOUR_WEB_CLIENT_ID.apps.googleusercontent.com">
   ```
3. Supabase Dashboard → **Authentication → Providers → Google** → toggle on → paste the **Client ID** and **Client Secret** from the Web application client.

> Google Sign-In on Flutter Web has known quirks (popup/FedCM behavior, the `signIn()` method being deprecated in favor of `renderButton`). If you hit issues there, prioritize testing on an actual Android device/emulator — it's the more stable, "real" path and matches your actual target platform.

## Step 7 — Run the app

```bash
flutter devices          # see what's available
flutter run -d <device_id>
```

For a first test, sign up with email/password — this requires zero additional setup and will let you verify the whole app end-to-end (profile, posting, feed, stories, explore, reels, activity, chat) once you have at least one post and, ideally, two test accounts to test follow/like/comment/message interactions between.

## Feature map (what's implemented, phase by phase)

| Area | What you get |
|---|---|
| **Auth** | Email/password sign up + sign in + password reset; Google Sign-In (Web client verified server-side by Supabase) |
| **Profile** | Avatar upload, bio/username/website editing, follower/following counts, posts grid, saved-posts tab, follow/unfollow, settings screen with dark/light/system theme toggle |
| **Home Feed** | Infinite scroll, stories row at top, double-tap-to-like with heart animation, comments bottom sheet, save/unsave, optimistic UI updates with rollback on failure |
| **Stories** | 24h auto-expiring stories (enforced both in queries and RLS), full-screen tap-through viewer with per-segment progress bars, view tracking, camera/gallery capture |
| **Create Post** | Camera or gallery (single photo, multi-photo carousel up to 10, or video), image cropping, caption with `#hashtag` parsing into a searchable hashtags table, location tag |
| **Explore** | Trending grid (most-liked posts in the last 7 days), fuzzy user search and hashtag search via Postgres `pg_trgm` indexes, hashtag detail grid |
| **Reels** | Vertical swipeable video feed, autoplay/pause based on scroll visibility, mute toggle, like/comment/save, video thumbnail generation for the profile grid |
| **Activity** | Realtime feed of new follows/likes/comments via Supabase Realtime, unread badge on the bottom nav, mark-as-read on open |
| **Direct Messaging** | 1-on-1 conversations (auto-created or reused), realtime message sync, text + image messages, unread counts per conversation, "New Message" user search to start a chat |
| **UI/UX polish included** | Light/dark/system theme (Instagram-style color palette), shimmer skeleton loaders for feed and stories while loading, pull-to-refresh throughout |

## Known gaps / good next steps

This is a strong, working v1 — but a few things are intentionally left as follow-ups rather than fully built, since they're either ops-heavy, platform-specific, or genuinely optional polish:

- **Apple Sign-In** was deliberately left out (this build targets Web + Android only). Re-adding it means: `sign_in_with_apple` package, an Apple Developer account, a Services ID, and Supabase's Apple provider config.
- **Video thumbnail generation** happens client-side at upload time via the `video_thumbnail` package — fine for this app's scale, but a server-side (Edge Function) approach would be more robust for very large videos.
- **Expired story cleanup** is handled at the *query* level (expired stories simply stop being fetched) and enforced by RLS, but the underlying Storage files for expired stories aren't automatically deleted. A scheduled Supabase Edge Function (cron) to purge expired rows + their Storage objects is a good addition before shipping to production.
- **Trending algorithm** in Explore is a simple "most-liked in the last 7 days" query. A real recommendation/decay-scoring system would live in a Supabase Edge Function or a scheduled job that pre-computes a trending score.
- **Group chat UI** — the database schema (`conversations.is_group`, `group_name`, `group_avatar_url`) fully supports group chats, and `chat_repository.dart` mostly assumes 1-on-1 in its `getOrCreateOneOnOneConversation` helper, but the UI flow for *creating* a group (picking multiple participants) isn't built yet — only 1-on-1 chat creation via `new_message_screen.dart`.
- **Post detail / single-post deep links** (e.g. tapping a grid thumbnail to see it full-screen with its own comments) is stubbed with empty `onTapPost` callbacks in a few places (`PostGrid`, Explore grid) — the data layer supports it, just needs a dedicated screen.
- **Comment replies** — the schema supports threaded replies (`comments.parent_id`), but the UI currently only shows a flat list.
- **Push notifications** (actual OS-level push, as opposed to in-app Activity feed) aren't implemented — this would require Firebase Cloud Messaging or a similar service layered on top of the existing `activities` table.

None of these block the app from running and being fully testable end-to-end — they're natural "Phase 12+" items.

## Troubleshooting

**`pubspec.yaml` dependency resolution fails mentioning `analyzer`/`freezed`/`custom_lint`** — this pubspec is already fixed to avoid this (no `riverpod_generator`/`custom_lint` packages, since nothing in this codebase uses `@riverpod` code-gen annotations). If you've modified `pubspec.yaml` and hit this, remove any linting/codegen packages you added and re-run `flutter pub get`.

**`build_runner` fails or produces no output** — make sure you're running it from the project root (not a subfolder), and that `flutter pub get` succeeded first.

**Missing asset directory errors on `flutter run`** — make sure `assets/images/` and `assets/icons/` exist (they're included in this zip, but if you restructured things, recreate them — even empty folders need a placeholder file like `.gitkeep` to survive some copy operations).

**Google Sign-In: "ClientID not set" on web** — you forgot to add the `<meta name="google-signin-client_id" ...>` tag to your generated `web/index.html`. See Step 6c.

**Google Sign-In: `invalid_client` / `Error 401`** — almost always means either (a) you're using the Android client ID somewhere it should be the Web client ID, or (b) your OAuth consent screen was never filled out/saved, or (c) you're testing with a Google account not on the Test Users list while the consent screen is in Testing mode.

**Google Sign-In: `popup_closed` on web** — Chrome's popup/FedCM/third-party-cookie behavior is finicky for `google_sign_in_web`. Prefer testing on an actual Android device/emulator; the web flow for this package is explicitly flagged as deprecated by the plugin itself in favor of `renderButton`.

**Gradle/Android build errors on Windows (Kotlin plugin warnings, `&&` not recognized in PowerShell, etc.)** — use `;` instead of `&&` to chain commands in PowerShell, and use `.\gradlew.bat` instead of `./gradlew` on Windows. Kotlin/AGP deprecation warnings during `flutter run` are almost always safe to ignore as long as the build reports `BUILD SUCCESSFUL`.

**Posts/stories don't appear after uploading** — double-check you ran the *entire* `0001_init.sql` file, including the storage bucket + storage policy sections at the bottom. Without the storage policies, uploads will fail silently or with a permissions error even though the buckets exist.

## A note on security

The Supabase **anon/public key** is safe to embed in this app's source — it's designed to be public. All actual data access control is enforced server-side via the Row Level Security (RLS) policies in `0001_init.sql`, not by hiding the key. Never embed your Supabase **service_role** key in client code; it bypasses RLS entirely and should only ever be used in trusted server environments.
